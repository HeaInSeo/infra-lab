# LAB_SCOPE

## 요약

`multipass-k8s-lab`은 범용 Kubernetes VM 랩 베이스라인입니다. 이 저장소의 역할은 Multipass 기반 Ubuntu 24.04 가상 머신 위에 반복 가능한 kubeadm 클러스터를 쉽게 생성, 삭제, 점검하고 가볍게 확장할 수 있게 만드는 것입니다.

이 저장소는 하나의 K8s PoC를 코드화하기 위한 곳이 아니라, 앞으로의 다양한 K8s PoC를 지원하기 위해 존재합니다.

## 목표

- 재사용 가능한 로컬 K8s VM 랩 기반 제공
- 클러스터 라이프사이클을 반복 가능하고 스크립트화된 형태로 유지
- Ubuntu 24.04 게스트 + Multipass + OpenTofu + kubeadm이라는 명확한 베이스라인에서 시작
- 첫 번째 실용적 랩 형태로 3노드 VM 클러스터 지원
- 로드밸런서, 로컬 스토리지, 향후 네트워킹 변형 같은 선택적 인프라 기능을 수용할 여지 확보

## 주요 사용 사례

- node-local artifact 및 local storage 실험
- DaemonSet 기반 노드 에이전트 실험
- same-node 재사용과 cross-node fetch 비교 실험
- Cilium 및 서비스 라우팅 검증
- 스토리지 동작 점검
- 오퍼레이터 개발 및 통합 테스트

## 저장소 책임 범위

- 필수 로컬 도구를 위한 호스트 부트스트랩 스크립트
- OpenTofu와 Multipass를 통한 VM 라이프사이클 오케스트레이션
- kubeadm 부트스트랩 및 조인 흐름
- 로컬 사용자를 위한 kubeconfig 내보내기
- 소규모 클러스터 지향 헬퍼 명령
- base 및 optional 인프라 애드온 관리
- 사용 경계에 대한 문서

## 비목표

- 애플리케이션 배포용 저장소가 아님
- artifact-agent, catalog, storage application, Cilium 구현 저장소가 아님
- 프로덕션급 클러스터 프로비저닝 도구가 아님
- 범용 멀티 프로바이더 추상화 계층이 아님
- 프로젝트 전용 워크로드를 베이스라인에 깊게 박아 넣는 장소가 아님

## 의도적으로 잘라낸 범위

`mac-k8s-multipass-terraform`과 비교했을 때, 이 저장소는 다음 항목을 의도적으로 제거하거나 피합니다.

- MySQL, Redis 헬퍼 같은 서비스별 VM 설정
- 서비스 스택 우선의 애드온 번들
- 랩 플랫폼 자체보다 과거 단일 테스트 흐름에 초점을 둔 문서
- 클러스터 bring-up과 프로젝트 전용 워크로드 사이의 암묵적 결합

## 애드온 정책

### Base

Base 애드온은 다음 조건을 모두 만족해야 합니다.

- 대부분의 랩 클러스터에 유용할 것
- 특정 프로젝트와의 결합도가 낮을 것
- 기본 베이스라인으로서 개념적 비용이 낮을 것

### Optional

Optional 애드온은 다음과 같은 경우 허용됩니다.

- 중요한 실험 범주를 열어 줄 것
- 의미 있는 클러스터 동작을 추가할 것
- 애플리케이션 지향이 아니라 인프라 지향을 유지할 것

## 프로필

`profiles/`는 앞으로 다음과 같은 오버레이를 위해 예약되어 있습니다.

- `cilium`
- `storage-lab`
- `operator-dev`
- `gateway-lab`

이 프로필들은 베이스라인을 다시 쓰는 대신, 베이스라인 위에 계층적으로 얹혀야 합니다.
